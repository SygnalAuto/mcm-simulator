## Summary

**Goal:** Add GitHub Actions CI that runs the pytest test suite across Python 3.11, 3.12, and 3.13 on every push to main and on pull requests targeting main.

**Architecture:** Single GitHub Actions workflow file using a Python version matrix. Uses pip to install the package with dev dependencies, then runs pytest.

**Tech Stack:** GitHub Actions, pip, pytest

## Scope

### In Scope

- GitHub Actions workflow file (`.github/workflows/ci.yml`)
- Python version matrix: 3.11, 3.12, 3.13
- Triggers: push to `main`, pull requests targeting `main`
- Test execution via pytest

### Out of Scope

- Linting / formatting (ruff) — can be added later
- Type checking (mypy/pyright) — no type annotations in codebase yet
- Coverage reporting
- Package publishing / release automation
- Branch protection rules (GitHub settings, not code)

## Approach

**Chosen:** Single workflow with Python version matrix

**Why:** Simple, standard GitHub Actions pattern. One file, easy to maintain. The version matrix catches compatibility issues across all supported Python versions at the cost of slightly longer total CI time (3 parallel jobs vs 1).

**Alternatives considered:**
- Separate workflows per version — unnecessary complexity for a small project with fast tests
- Nox/tox test runner — adds a dependency for no benefit when the test command is just `pytest`

## Context for Implementer

> Write for an implementer who has never seen the codebase.

- **Project type:** Python package using setuptools, declared in `pyproject.toml`
- **Python requirement:** `>=3.11` (from `pyproject.toml:9`)
- **Dependencies:** `python-can>=4.3.0`, `cantools>=39.0.0` (runtime); `pytest>=8.0` (dev, in `[project.optional-dependencies] dev`)
- **Test command:** `pytest` (no special flags needed)
- **Test speed:** 42 tests, ~0.9s — very fast
- **DBC files:** `.dbc` files in `mcm_simulator/dbc/` are package data, loaded at runtime by tests — they're in the repo so no special CI handling needed
- **No existing CI:** No `.github/` directory exists yet

## Assumptions

- GitHub-hosted runners have access to install `python-can` and `cantools` from PyPI — supported by both being pure-Python or having pre-built wheels for Linux. All 42 tests depend on this.
- `ubuntu-latest` runner is sufficient — no platform-specific (Windows/macOS) testing needed. Task 1 depends on this.

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| `python-can` or `cantools` install fails on a specific Python version | Low | Blocks that matrix entry | Pin minimum versions already in pyproject.toml; CI will surface the failure immediately |
| Tests rely on CAN hardware not present in CI | Low | All tests fail | Verified locally — tests use mock CAN interfaces, no real hardware needed |
